# .PLAN-v0.md — Running Agenticana in GitHub Actions Like Pi

> **Version:** 0.1.0 | **Date:** 2026-03-13
> **Goal:** Use the Agenticana source to run as a GitHub Actions–powered AI agent, modeled after how [github-minimum-intelligence](https://github.com/japer-technology/github-minimum-intelligence) uses `pi` (pi-mono).

---

## Executive Summary

**github-minimum-intelligence** proves that a single GitHub repository can host a fully autonomous AI agent using only Issues (UI), Actions (compute), and Git (memory). It runs the [`pi` coding agent](https://github.com/badlogic/pi-mono) inside a GitHub Actions workflow triggered by issue events, committing all conversation state back to the repo.

**Agenticana** is a 20-agent, multi-skill AI system currently designed for local VS Code + Copilot usage. This plan describes how to adapt Agenticana's agents, skills, router, ReasoningBank, and MCP tools to execute inside GitHub Actions — making it a repository-native AI workforce that responds to Issues, just like `pi` does in github-minimum-intelligence.

## How github-minimum-intelligence Works (Reference Architecture)

### The Closed Loop

```
User opens issue / comments
    → GitHub Actions workflow triggers
    → Authorization check (collaborator with write+ access)
    → 🚀 reaction added as "working" indicator
    → Bun runtime installed, dependencies cached
    → agent.ts orchestrator runs:
        1. Reads issue title/body from webhook payload
        2. Resolves or creates a session mapping (issue # → session file)
        3. Builds prompt from event context
        4. Validates LLM provider API key
        5. Spawns `pi` agent with --session for continuity
        6. Extracts reply from JSONL output
        7. Persists session mapping
        8. Commits all state to git (with retry-on-conflict)
        9. Posts reply as issue comment
    → 👍 reaction on success, 👎 on error
```

### Key Primitives

| GitHub Primitive | System Role |
|---|---|
| **Issues** | Conversational UI — each issue is a persistent AI thread |
| **Actions** | Compute runtime — ephemeral 2-core/7GB runners |
| **Git commits** | Memory — session transcripts, state, and edits are versioned |
| **Secrets** | API key storage — LLM provider keys via Settings → Secrets |
| **Reactions** | Status indicators — 🚀 (working), 👍 (success), 👎 (error) |
| **Permissions** | Access control — write+ collaborators only |

### File Layout

```
.github-minimum-intelligence/
├── .pi/                        # Agent personality & skills config
│   ├── settings.json           # LLM provider, model, thinking level
│   ├── APPEND_SYSTEM.md        # System prompt loaded every session
│   ├── BOOTSTRAP.md            # First-run identity prompt
│   └── skills/                 # Modular skill packages
├── lifecycle/
│   └── agent.ts                # Core orchestrator (~500 lines)
├── install/                    # Default templates for fresh installs
├── state/
│   ├── issues/N.json           # Issue # → session file mapping
│   └── sessions/*.jsonl        # Full conversation transcripts
├── docs/                       # Documentation
├── public-fabric/              # GitHub Pages output
├── AGENTS.md                   # Agent identity file
├── VERSION                     # Installed version (semver)
├── package.json                # Runtime: @mariozechner/pi-coding-agent
└── bun.lock                    # Locked dependencies
```

### Workflow Structure (Three Jobs)

1. **run-install** — Self-installer triggered by `workflow_dispatch`. Downloads the agent folder from the template repo, handles fresh install vs upgrade with user file preservation.
2. **run-agent** — Core AI agent triggered by `issues.opened` and `issue_comment.created`. Runs authorization, installs deps, executes `agent.ts`.
3. **run-gitpages** — Publishes `public-fabric/` to GitHub Pages on push to main.

#### 0.2 Create the Workflow YAML

File: `.github/workflows/github-agenticana-agent.yml`

```yaml
name: github-agenticana-agent

on:
  issues:
    types: [opened]
  issue_comment:
    types: [created]
  push:
    branches: ["main"]
    paths-ignore:
      - ".github/workflows/**"
  workflow_dispatch:

permissions:
  contents: write
  issues: write
  actions: write
  pages: write
  id-token: write

jobs:
  run-install:
    # Self-installer — downloads .github-agenticana/ from template repo
    runs-on: ubuntu-latest
    if: github.event_name == 'workflow_dispatch'
    steps:
      - uses: actions/checkout@v6
      - name: Install from template
        run: |
          # Download and extract .github-agenticana/ folder
          # Handle fresh install vs upgrade with backup/restore
          # ...

  run-agent:
    runs-on: ubuntu-latest
    concurrency:
      group: agenticana-${{ github.repository }}-issue-${{ github.event.issue.number }}
      cancel-in-progress: false
    if: >-
      (github.event_name == 'issues')
      || (github.event_name == 'issue_comment' && !endsWith(github.event.comment.user.login, '[bot]'))
    steps:
      - name: Authorize
        # Check collaborator permissions (write+)
      - name: Checkout
        uses: actions/checkout@v6
        with:
          fetch-depth: 0
      - name: Setup Bun
        uses: oven-sh/setup-bun@v2
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: |
          cd .github-agenticana && bun install --frozen-lockfile
          pip install sentence-transformers
      - name: Run Agent
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: bun .github-agenticana/lifecycle/agent.ts
```

### Phase 1 — Intelligent Routing (Multi-Agent via Labels or Content)

**Goal:** Route incoming issues to the correct specialist agent based on content analysis or labels.

#### 1.1 Issue Label → Agent Mapping

Users can apply labels to issues to explicitly select an agent:

| Label | Agent | Description |
|---|---|---|
| `agent:frontend` | `frontend-specialist` | React, Next.js, UI, CSS |
| `agent:backend` | `backend-specialist` | APIs, Node.js, server logic |
| `agent:debug` | `debugger` | Bug fixes, error analysis |
| `agent:security` | `security-auditor` | Auth reviews, vulnerability checks |
| `agent:database` | `database-architect` | Prisma, SQL, migrations |
| `agent:devops` | `devops-engineer` | Docker, CI/CD, deploy |
| `agent:test` | `test-engineer` | Unit tests, E2E tests |
| `agent:plan` | `project-planner` | Architecture planning |
| `agent:orchestrate` | `orchestrator` | Complex multi-domain tasks |
| *(no label)* | Auto-detect | Content analysis selects agent |

#### 1.2 Content-Based Auto-Routing

When no agent label is present, `agent.ts` analyzes the issue content using Agenticana's existing routing logic:

```typescript
// In agent.ts, before spawning pi:
const routerResult = await runRouter(prompt);
// routerResult = { model: "pro", tier: "MODERATE", agent: "frontend-specialist", skills: [...] }
```

This reuses the `router/complexity-scorer.js` and `router/router.js` that already exist in the Agenticana source.

#### 1.3 Dynamic System Prompt Construction

Based on the selected agent, construct the system prompt:

```
[GEMINI.md rules (always)]
[Selected agent .md file]
[Tier 1 skills (always)]
[Tier 2 skills (if domain match)]
[Tier 3 skills (if explicitly needed)]
[ReasoningBank context (if similarity > 0.60)]
```

#### 2.2 Pre-Execution: ReasoningBank Query

Before every agent run, query the ReasoningBank:

```typescript
// Phase -1: Check for similar past decisions
const bankResult = await runReasoningBankQuery(prompt);
if (bankResult.similarity >= 0.85) {
  // Fast path — include past decision in prompt as reference
  systemPrompt += `\n\n## Past Decision (similarity: ${bankResult.similarity})\n${bankResult.decision}`;
}
```

#### 2.3 Post-Execution: Record Decision

After the agent completes, record the decision:

await recordDecision({

});

### Phase 3 — Multi-Agent Orchestration (Swarm in Actions)

**Goal:** Enable the orchestrator agent to dispatch sub-tasks to specialist agents within a single workflow run.

#### 3.1 Sequential Agent Invocation

For complex multi-domain tasks (detected by the orchestrator), the workflow can invoke multiple agents sequentially within a single run:

```typescript
// orchestrator detects multi-domain task
const plan = await generatePlan(prompt);
// plan = [
//   { agent: "database-architect", task: "Design the schema" },
//   { agent: "backend-specialist", task: "Build the API endpoints" },
//   { agent: "frontend-specialist", task: "Create the UI components" },
//   { agent: "test-engineer", task: "Write integration tests" },
// ]

for (const step of plan) {
  const result = await runAgentStep(step.agent, step.task, sessionPath);
  results.push(result);
}
```

#### 3.2 Issue Threading

Each sub-task can be posted as a separate comment on the issue, creating a visible execution thread:

```
Issue #42: "Build a user dashboard with auth"
  └─ Comment 1: 🤖 Orchestrator — Analyzing task, 4 agents needed...
  └─ Comment 2: 🗄️ database-architect — Schema designed: users, sessions, preferences
  └─ Comment 3: ⚙️ backend-specialist — API endpoints: /api/users, /api/sessions, /api/prefs
  └─ Comment 4: 🎨 frontend-specialist — Components: Dashboard, UserProfile, AuthForm
  └─ Comment 5: 🧪 test-engineer — 23 tests written, all passing
  └─ Comment 6: ✅ Orchestrator — Task complete. Summary: [...]
```

#### 3.3 Parallel Agent Dispatch (Advanced)

For independent sub-tasks, use GitHub Actions matrix strategy:

```yaml
jobs:
  plan:
    # Orchestrator creates the plan
    outputs:
      agents: ${{ steps.plan.outputs.agents }}
  execute:
    needs: plan
    strategy:
      matrix:
        agent: ${{ fromJson(needs.plan.outputs.agents) }}
    steps:
      - run: bun .github-agenticana/lifecycle/agent.ts --agent ${{ matrix.agent }}
  merge:
    needs: execute
    # Merge results and post summary
```

## Cost Model

Based on the [github-actions-cost-model analysis](.github-agenticana/githubification/github-actions-cost-model.md):

| Scenario | Actions Minutes/Month | Actions Cost | LLM Cost (est.) | Total |
|---|---|---|---|---|
| **Light** (5 interactions/day) | ~150 min | Free tier | ~$15-30 | ~$15-30 |
| **Moderate** (20/day) | ~600 min | Free tier | ~$60-120 | ~$60-120 |
| **Heavy** (50/day) | ~1,500 min | ~$4 overage | ~$150-300 | ~$154-304 |
| **Intense** (100+/day) | ~3,000 min | ~$16 overage | ~$300-600 | ~$316-616 |

> **Key insight:** LLM API costs dominate (60-80%). Actions minutes are cheap. Agenticana's model router saves 30-60% on LLM costs by routing simple tasks to cheaper models — a significant advantage over MI's single-model approach.

## Hybrid Model (Recommended)

The optimal setup runs both models simultaneously:

```
                    ┌──────────────────────────┐
                    │    GitHub Repository      │
                    │                           │
  ┌─────────────┐  │  .github-agenticana/      │  ┌──────────────┐
  │  VS Code    │  │  ├── agents/              │  │  GitHub       │
  │  + Copilot  │──│  ├── skills/              │──│  Issues       │
  │  (local)    │  │  ├── state/ (git)         │  │  (remote)     │
  └─────────────┘  │  ├── memory/ (git)        │  └──────────────┘
        │          │  └── lifecycle/agent.ts    │         │
        │          └──────────────────────────┘         │
        ▼                                                ▼
  Local Copilot Chat                              GitHub Actions
  (interactive dev)                               (async agent)
        │                                                │
        └───── Shared state via git pull/push ───────────┘
```

- **Local mode:** Developer uses Copilot Chat with MCP tools for interactive work.
- **Actions mode:** Team members open Issues for async agent assistance.
- **Both share:** The same agents, skills, ReasoningBank, and state via git.
